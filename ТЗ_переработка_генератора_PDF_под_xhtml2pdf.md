# Техническое задание

## Переработка генератора заявления на возврат: переход с Chromium/Playwright на xhtml2pdf

### 1. Цель переработки

Переработать существующий Python-скрипт `refund_application_generator.py`, чтобы PDF формировался из HTML-шаблона библиотекой `xhtml2pdf`, без Chromium и Playwright.

Целевая операционная система — Windows 10 и Windows 11. Microsoft Word, Chromium, Edge WebDriver, Playwright, LibreOffice и другие внешние программы для создания PDF не использовать.

Для формирования заявления использовать новый шаблон:

```text
templates/refund_application_template_xhtml2pdf.html
```

Бизнес-логика чтения JSON, проверки данных, расчёта суммы, формирования имени файла, логирования и открытия готового PDF должна сохраниться.

### 2. Исходное состояние

В существующей реализации функция `create_pdf()` запускает Chromium через Playwright и передаёт браузеру заполненный HTML. Текущий HTML-шаблон использует CSS, рассчитанный на браузерный движок, в том числе CSS Grid и экранные/печатные медиазапросы.

После переработки:

- `playwright` не должен импортироваться или запускаться;
- установка браузера командой `playwright install chromium` не требуется;
- процессы Chromium не создаются;
- PDF создаётся вызовом `xhtml2pdf.pisa.CreatePDF()`;
- используется HTML-шаблон, специально перевёрстанный под возможности `xhtml2pdf`.

Разрешается удалить старый браузерный шаблон только после успешной интеграции и проверки нового. Если он сохраняется в репозитории, программа не должна использовать его по умолчанию.

### 4. Зависимости

В `requirements.txt`:

1. удалить `playwright`;
2. добавить фиксированную совместимую версию `xhtml2pdf`;
3. не добавлять отдельный браузер или исполняемый PDF-конвертер.

Пример:

```text

```

После обычной установки зависимостей команда формирования PDF должна работать без дополнительных команд загрузки браузера:

```powershell
python -m pip install -r requirements.txt
```

### 5. Конфигурация

В `config.ini` изменить путь к шаблону:

```ini
[paths]
template = templates/refund_application_template_xhtml2pdf.html
logs_dir = logs
pdf_output_dir = output
```

Относительные пути, как и раньше, разрешать относительно каталога `config.ini`, а не относительно текущего рабочего каталога процесса.

Шрифты должны находиться в каталоге `fonts` рядом с каталогом `templates`. Их расположение задано в CSS шаблона относительными путями `../fonts/...`.

При отсутствии шаблона или одного из шрифтов завершать работу до создания PDF с понятным сообщением на русском языке и записью ошибки в лог.

### 6. Требования к HTML-шаблону

Использовать приложенный `refund_application_template_xhtml2pdf.html` без обратной замены таблиц на CSS Grid или Flexbox.

Шаблон должен:

- использовать HTML-таблицы для блоков подписей и правой шапки;
- использовать только поддерживаемые `xhtml2pdf` CSS-свойства;
- задавать A4 и поля страницы через `@page`;
- подключать локальные DejaVu Serif TTF-файлы через `@font-face`;
- сохранять линии для ручного заполнения;
- содержать метки `{recipient}`, `{org}`, `{address}`, `{date}`, `{items}`, `{total_amount}`, `{application_date}`;
- не содержать JavaScript, медиазапросы, CSS Grid, Flexbox, `box-sizing`, `white-space: pre-line`, `min-height` и браузерные экранные стили.

Исходный HTML-шаблон не изменять во время выполнения программы. Заполненный документ формировать в памяти.

### 7. Подстановка данных

Правила заполнения остальных меток сохранить без изменений:

| Метка | Источник |
| --- | --- |
| `{recipient}` | `recipient` из JSON |
| `{org}` | `org` из JSON |
| `{address}` | `address` из JSON |
| `{date}` | дата покупки из `date`, преобразованная из `YYYYMMDD` в `DD.MM.YYYY` |
| `{items}` | наименования из `items[].name` |
| `{total_amount}` | сумма всех `items[].price` в целых рублях |
| `{application_date}` | текущая системная дата в формате `DD.MM.YYYY` |

Все строковые значения из JSON сначала экранировать функцией `html.escape(..., quote=True)`.

Для `{items}` сформировать контролируемый HTML-фрагмент:

- каждое наименование сначала экранировать отдельно;
- один товар выводить без номера;
- несколько товаров выводить по одному на строку с нумерацией;
- строки оформлять как `<div class="item">...</div>`;
- не экранировать повторно уже собранные программой теги `<div>`;
- никакой HTML из значения `items[].name` не должен попадать в документ как разметка.

Пример безопасного результата для двух товаров:

```html
<div class="item">1. Первый товар</div>
<div class="item">2. Второй товар</div>
```

Функцию проверки меток сохранить. После подстановки в HTML не должно оставаться неизвестных или незаполненных меток вида `{...}`.

### 8. Реализация create_pdf()

Заменить текущую реализацию `create_pdf()` на формирование PDF через `xhtml2pdf`.

Рекомендуемый интерфейс:

```python
def create_pdf(document_html: str, pdf_path: Path, template_path: Path) -> None:
    ...
```

Функция должна:

1. лениво импортировать `pisa` из `xhtml2pdf` либо импортировать его на уровне модуля;
2. открыть итоговый PDF в бинарном режиме `wb`;
3. вызвать `pisa.CreatePDF()` с UTF-8 HTML, файлом назначения и базовым путём шаблона;
4. корректно разрешить относительные ссылки на TTF-файлы;
5. проверить результат конвертации по `status.err`;
6. проверить, что PDF создан, имеет ненулевой размер и начинается с сигнатуры `%PDF-`;
7. при любой ошибке удалить неполный PDF;
8. преобразовать техническую ошибку в `ApplicationError` с понятным русским сообщением.

Для ресурсов реализовать `link_callback(uri, rel)` либо эквивалентную функцию. Она должна:

- разрешать только локальные файлы проекта;
- преобразовывать относительный URI шрифта в абсолютный путь;
- запрещать `http://`, `https://` и другие сетевые URI;
- проверять существование ресурса;
- не позволять выйти за корневой каталог проекта через `..` после нормализации пути.

Пример вызова:

```python
status = pisa.CreatePDF(
    src=document_html,
    dest=pdf_file,
    encoding="utf-8",
    path=template_path.resolve().as_uri(),
    link_callback=link_callback,
)
```

Если конкретная версия `xhtml2pdf` ожидает файловый путь, а не URI в параметре `path`, использовать документированный для закреплённой версии вариант и покрыть его интеграционным тестом.

### 9. Изменения основного сценария

В месте вызова `create_pdf()` дополнительно передать путь исходного HTML-шаблона.

Сохранить существующий порядок:

1. загрузить `config.ini`;
2. настроить логирование;
3. загрузить и проверить JSON;
4. прочитать HTML-шаблон как UTF-8/UTF-8 BOM;
5. проверить наличие обязательных меток;
6. безопасно подставить значения;
7. сформировать свободное имя PDF;
8. создать PDF через `xhtml2pdf`;
9. записать успешный результат в лог;
10. вывести полный путь к PDF;
11. открыть PDF через `os.startfile()`.

Ошибка открытия уже успешно созданного PDF не должна удалять файл. Пользователю вывести путь и отдельное сообщение о том, что автоматическое открытие не удалось.

### 10. Обработка ошибок и логирование

Добавить понятные ошибки как минимум для случаев:

В логах не сохранять персональные данные и полный заполненный HTML. Разрешено записывать путь шаблона, путь результата, количество товаров, сумму, длительность конвертации и технический текст исключения.

### 13. Критерии приёмки

Переработка считается выполненной, если:

1. заявление создаётся на Windows 10/11 после обычного `pip install -r requirements.txt`;
2. Chromium, Playwright, Microsoft Word и внешние PDF-конвертеры не требуются;
3. используется `refund_application_template_xhtml2pdf.html`;
4. русский текст корректно отображается и извлекается из PDF;
5. внешний вид соответствует исходному заявлению: A4, правая шапка, линии, два основания возврата и три блока подписей;
6. данные берутся из тех же полей JSON и сумма считается по прежним правилам;
7. личные сведения покупателя остаются пустыми для ручного заполнения;
8. шаблон и ранее созданные PDF не изменяются и не перезаписываются;
9. неполный PDF не остаётся после ошибки конвертации;
10. готовый PDF автоматически открывается стандартной программой Windows;
11. `requirements.txt`, `README.MD`, `config.ini` и тесты соответствуют новой реализации;
12. в исходном коде и документации нет команд установки Chromium и рабочих импортов Playwright.
