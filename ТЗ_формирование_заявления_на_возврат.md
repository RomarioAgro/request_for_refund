# Техническое задание

## Автоматическое формирование заявления на возврат в PDF

### 1. Назначение

Необходимо разработать Python-скрипт, который по данным из JSON-файла автоматически формирует заполненное заявление на возврат в формате PDF и затем открывает созданный PDF в установленной по умолчанию программе просмотра.

Целевая операционная система — Windows 10 и Windows 11.

За основу оформления использовать готовый HTML-шаблон `templates/refund_application_template.html`. Этот HTML-файл является исходным и единственным шаблоном заявления.

Преобразовывать шаблон из DOC/DOCX, Markdown или другого формата не требуется.

Пример входного файла: `392250_01_request_for_refund(1).json`.

### 2. Общий сценарий работы

1. Пользователь запускает скрипт и передаёт ему путь к JSON-файлу возврата.
2. Скрипт загружает настройки из файла `config.ini`.
3. Скрипт читает и проверяет JSON.
4. Скрипт загружает HTML-шаблон заявления, путь к которому задан в `config.ini`.
5. В HTML подставляются сведения, доступные для автоматического заполнения.
6. Рассчитывается общая сумма возврата по всем товарам из JSON.
7. Заполненное заявление преобразуется в PDF.
8. PDF сохраняется в папку, заданную в `config.ini`.
9. Сразу после успешного формирования скрипт открывает созданный PDF в системной программе просмотра PDF.

Никакие дополнительные формы ввода и кнопки перед созданием заявления не требуются. PDF формируется автоматически при вызове скрипта.

### 3. Входные данные

Скрипт должен принимать путь к одному JSON-файлу возврата.

Ожидаемая структура данных:

```json
{
  "date": "20260720",
  "org": "ООО \"Клевер Урал\"",
  "recipient": "Генеральному директору ООО \"Клевер Урал\" Валейня И. Л.",
  "address": "Тольятти, Автозаводское шоссе, 6",
  "items": [
    {
      "name": "Наименование товара 1",
      "price": 2932
    },
    {
      "name": "Наименование товара 2",
      "price": 1732
    }
  ]
}
```

Скрипт должен корректно читать русские символы. Необходимо поддержать как минимум UTF-8/UTF-8 BOM и Windows-1251 либо автоматически определить одну из этих кодировок.

### 4. Соответствие полей JSON полям заявления

| Данные в заявлении | Источник | Метка шаблона | Правило заполнения |
| --- | --- | --- | --- |
| Получатель заявления | `recipient` | `{recipient}` | Подставить без изменения текста |
| Наименование организации/магазина | `org` | `{org}` | Подставить без изменения текста |
| Адрес магазина | `address` | `{address}` | Подставить без изменения текста |
| Дата покупки | `date` | `{date}` | Преобразовать из формата `YYYYMMDD` в формат `DD.MM.YYYY` |
| Возвращаемые товары | `items[].name` | `{items}` | Для каждого товара брать из JSON только наименование |
| Общая сумма возврата | `items[].price` | `{total_amount}` | Сложить цены всех элементов массива `items` |
| Дата заявления | Системная дата компьютера | `{application_date}` | Использовать дату формирования PDF в формате `DD.MM.YYYY` |

Поле `date` из JSON является датой покупки, а не датой заявления. Значение этого поля необходимо подставлять в шаблон в метку `{date}`. Дата заявления всегда должна быть текущей системной датой и подставляться в отдельную метку `{application_date}`.

### 5. Вывод товаров

1. Если возвращается один товар, вывести его наименование в предназначенной для товара строке заявления.
2. Если возвращается несколько товаров, вывести наименование каждого товара отдельно, в порядке следования в массиве `items`.
3. Для каждого товара из JSON использовать только поле `name`.
4. Артикул, код товара, количество, цену отдельного товара и сумму отдельной позиции в заявлении не выводить.
5. Если наименований несколько, допускается нумерованный список, если он лучше сохраняет читаемость документа.

### 6. Расчёт и вывод суммы

1. Единственный источник суммы — значения `price` в массиве `items` входного JSON.
2. Общая сумма рассчитывается так:

```text
total = сумма всех items[i].price
```

3. Другие источники суммы, скидки, количества товаров и дополнительные перерасчёты не использовать.
4. Все суммы считаются целыми рублями. Расчёты с копейками и правила округления не требуются.
5. В строке «По цене» вывести общую сумму возврата по всем товарам цифрами, без записи суммы прописью.
6. Формат вывода: разделять разряды пробелом и указывать валюту, например:

```text
По цене: 7 596 рублей.
```

7. Для приложенного примера расчёт должен дать:

```text
2 932 + 1 732 + 2 932 = 7 596 рублей
```

### 7. Данные, заполняемые покупателем вручную

Программа не должна запрашивать и автоматически подставлять Ф. И. О. покупателя и прочие личные сведения покупателя.

Все предусмотренные образцом поля, данных для которых нет в JSON, необходимо оставить в виде пустых линий достаточной длины для заполнения от руки. Покупатель заполняет их самостоятельно после печати заявления.

Подписи также всегда остаются пустыми и ставятся вручную.

### 8. Работа с шаблоном

1. Неизменяемый текст, CSS-стили и оформление заявления должны храниться в готовом HTML-шаблоне `refund_application_template.html`.
2. Не хранить полный текст заявления внутри Python-кода.
3. В шаблоне использовать однозначные метки подстановки, например:

```text
{recipient}
{org}
{address}
{date}
{items}
{total_amount}
{application_date}
```

4. Все метки без исключения должны иметь единый формат: имя поля в одной паре фигурных скобок — `{field_name}`. Двойные фигурные скобки, угловые скобки и смешение разных форматов не использовать.
5. Метка `{date}` зарезервирована именно для даты покупки из JSON. Для даты формирования заявления необходимо использовать отдельную метку `{application_date}`.

6. Исходный HTML-шаблон во время работы скрипта не изменять. Для каждого заявления формировать HTML в памяти или во временном файле.
7. Все значения, полученные из JSON, перед подстановкой обязательно экранировать как HTML-текст. Они не должны интерпретироваться как HTML-код или JavaScript.
8. Для метки `{items}` сформировать одну текстовую строку на каждый товар, разделяя товары переводом строки. CSS-свойство `white-space: pre-line` должно сохранять эти переносы без вставки HTML-тегов из входных данных.
9. После подстановки данных в итоговом PDF не должно оставаться служебных меток вида `{...}`.
10. Строку образца с суммой необходимо адаптировать под целые рубли: поле копеек в итоговом заявлении не выводить.
11. HTML-шаблон является единственным источником разметки документа. Генератор не должен читать, создавать или преобразовывать шаблоны DOC, DOCX либо Markdown.

### 9. Требования к PDF

1. Формат страницы — A4.
2. Сохранить шрифты, выравнивание, отступы и общий внешний вид предоставленного образца.
3. Текст не должен выходить за поля страницы, перекрываться или обрезаться.
4. Документ должен корректно печататься на обычном принтере.
5. При увеличении количества товаров допускается создание нескольких страниц.
6. Итоговая сумма, строки подписи и относящиеся к ним пояснения не должны обрезаться или накладываться друг на друга.
7. Сформированный PDF должен открываться стандартной программой просмотра PDF, установленной в операционной системе.
8. PDF формировать из заполненного HTML средствами Chromium через библиотеку Playwright для Python.
9. Microsoft Word, COM-интерфейс, `pywin32` и процесс `WINWORD.EXE` не использовать. Microsoft Word не является системной зависимостью.
10. При формировании PDF использовать формат A4, печатные CSS-стили, масштаб 100 %, включённую печать фоновой графики и поля, заданные правилом CSS `@page`.
11. Не добавлять в PDF стандартные колонтитулы браузера: URL, название страницы, дату печати и номера страниц.
12. Временный заполненный HTML после успешного создания PDF удалить. При ошибке допустимо сохранить его для диагностики, зафиксировав путь в логе.

### 10. Конфигурация

В корне проекта обязательно должен находиться файл `config.ini`. В нём должны задаваться:

- путь к шаблону заявления;
- путь к папке логов;
- путь к папке, в которую сохраняются готовые PDF.

Пример `config.ini`:

```ini
[paths]
template = templates/refund_application_template.html
logs_dir = logs
pdf_output_dir = output
```

Требования к работе с конфигурацией:

1. Все относительные пути из `config.ini` разрешать относительно каталога, в котором находится сам `config.ini`, а не относительно текущего рабочего каталога процесса.
2. Не размещать пути к шаблону, логам и PDF непосредственно в Python-коде.
3. Если `config.ini` отсутствует, недоступен для чтения, не содержит секцию `[paths]` или обязательный параметр, скрипт должен завершиться с понятным сообщением на русском языке.
4. Если папка логов или папка для PDF отсутствует, скрипт должен создать её автоматически.
5. Если путь к шаблону неверен или файл шаблона отсутствует, PDF не формировать.

### 11. Имена и хранение файлов

Рекомендуемая структура проекта:

```text
project/
├── templates/
│   └── refund_application_template.html
├── logs/
├── output/
├── config.ini
├── refund_application_generator.py
└── requirements.txt
```

В `requirements.txt` должна быть указана библиотека `playwright`. После установки зависимостей необходимо один раз установить браузер Chromium командой `playwright install chromium` либо включить браузер в дистрибутив приложения.

Имя PDF формировать по шаблону:

```text
Заявление_на_возврат_<имя_исходного_JSON>_<YYYY-MM-DD_HH-MM-SS>.pdf
```

Пример:

```text
Заявление_на_возврат_392250_01_request_for_refund_2026-07-21_16-30-05.pdf
```

Добавление времени обязательно, чтобы повторный запуск не перезаписывал ранее созданный файл.

### 12. Проверка входных данных и ошибки

Перед формированием заявления проверить:

- файл существует и доступен для чтения;
- содержимое является корректным JSON;
- корневой элемент JSON является объектом;
- присутствуют непустые поля `recipient`, `org`, `address` и `date`;
- поле `date` содержит корректную дату покупки в формате `YYYYMMDD`;
- поле `items` существует, является массивом и содержит хотя бы один товар;
- у каждого товара есть непустое строковое поле `name`;
- у каждого товара есть поле `price` с допустимым целым числовым значением;
- цена не является отрицательной;
- рассчитанная общая сумма больше нуля;
- файл `config.ini` существует и содержит все обязательные настройки;
- файл шаблона существует;
- каталог логов доступен для записи либо может быть создан;
- каталог результатов доступен для записи;
- преобразование в PDF завершилось успешно и итоговый PDF существует.

При ошибке вывести понятное сообщение на русском языке с именем проблемного файла или поля. Повреждённый либо неполный JSON не должен приводить к созданию частично заполненного заявления.

Приложенный пример JSON в текущем виде синтаксически некорректен: после значения `address` отсутствует запятая, а после значений `price` стоят лишние запятые. До использования файл должен быть исправлен либо источник JSON должен формировать валидный JSON.

### 13. Логирование

В лог записывать только технические сведения:

- время запуска;
- путь к входному JSON;
- путь к шаблону;
- количество товаров;
- рассчитанную общую сумму;
- путь к созданному PDF;
- сведения об ошибках.

Личные сведения покупателя в лог не записывать.

Логи сохранять в папку, указанную параметром `logs_dir` файла `config.ini`.

### 14. Требования к запуску

Основной вариант запуска:

```text
python refund_application_generator.py "путь_к_файлу.json"
```

Скрипт предназначен для запуска в Windows. Открытие готового файла выполнять штатным способом Windows, например через `os.startfile()`, чтобы использовалась назначенная пользователем программа просмотра PDF.

После успешного выполнения скрипт обязан:

1. сохранить PDF;
2. вывести в консоль полный путь к созданному файлу;
3. автоматически открыть этот PDF.

### 15. Критерии приёмки

Реализация считается готовой, если выполняются все условия:

1. Скрипт запускается с указанием пути к JSON.
2. Получатель, организация и адрес берутся из соответствующих полей JSON.
3. Из каждой позиции товара выводится только наименование из `items[].name`.
4. При нескольких товарах все их наименования присутствуют в заявлении.
5. В строке «По цене» указана сумма всех `items[].price`.
6. Сумма отображается цифрами и в целых рублях, без суммы прописью и без копеек.
7. Дата покупки берётся из поля `date` JSON и подставляется в метку `{date}`.
8. Дата заявления совпадает с системной датой формирования PDF и не заменяется датой покупки.
9. Ф. И. О. и личные данные покупателя не запрашиваются и не подставляются; вместо них сохранены линии для ручного заполнения.
10. В PDF не осталось служебных меток шаблона.
11. PDF имеет формат A4, текст читаем, ничего не обрезано.
12. Ранее созданные PDF не перезаписываются.
13. После создания PDF автоматически открывается.
14. Путь к шаблону, папке логов и папке PDF берётся из `config.ini`.
15. Логи и PDF создаются в папках, заданных в `config.ini`.
16. При некорректном JSON или конфигурации выводится понятная ошибка, а PDF не создаётся.
17. Проект работает под Windows 10 и Windows 11 без установленного Microsoft Word.
18. HTML преобразуется в PDF через Chromium/Playwright; стандартные колонтитулы браузера в PDF отсутствуют.
19. Все метки шаблона имеют единый формат `{field_name}`; метки с двойными фигурными скобками не используются.
20. Для генерации используется непосредственно `refund_application_template.html`; промежуточное преобразование шаблона из DOC, DOCX или Markdown отсутствует.

### 16. Минимальные тестовые сценарии

1. Один товар с целой ценой.
2. Несколько товаров; проверка суммы и порядка наименований.
3. Товар с длинным наименованием.
4. Большое количество товаров, приводящее к переносу на следующую страницу.
5. JSON в UTF-8 с русским текстом.
6. JSON в Windows-1251 с русским текстом.
7. Отсутствует одно из обязательных полей верхнего уровня.
8. Пустой массив `items`.
9. У товара отсутствует `name` или `price`.
10. Цена отрицательная или не является целым числом.
11. Повреждённый JSON.
12. Повторный запуск для одного и того же JSON: оба PDF сохранены под разными именами.
13. Успешное автоматическое открытие созданного PDF.
14. Поле `date` содержит корректную дату покупки: дата правильно преобразована и подставлена в `{date}`.
15. Поле `date` отсутствует или содержит невозможную дату: PDF не создаётся, выведена понятная ошибка.
16. Пути в `config.ini` относительные: они корректно разрешаются относительно расположения файла конфигурации.
17. Папки логов и PDF из `config.ini` отсутствуют: они создаются автоматически.
18. В `config.ini` отсутствует один из обязательных параметров: скрипт завершается с понятной ошибкой.
19. Chromium для Playwright не установлен или не запускается: PDF не создаётся, выводится понятная ошибка на русском языке с инструкцией `playwright install chromium`.
20. Значения JSON содержат символы `<`, `>`, `&`, кавычки или фигурные скобки: они безопасно отображаются как текст и не ломают HTML.
21. После успешной и неуспешной конвертации процесс Chromium, запущенный скриптом, корректно закрывается.
