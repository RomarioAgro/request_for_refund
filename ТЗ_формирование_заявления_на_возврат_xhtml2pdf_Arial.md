# Техническое задание

## Формирование заявления на возврат в PDF из HTML-шаблона через xhtml2pdf

## 1. Назначение

Необходимо переработать существующее Windows-приложение `refund_application_generator.py`, которое получает путь к JSON-файлу возврата, заполняет заявление и сохраняет его в PDF.

PDF должен формироваться непосредственно из готового HTML-шаблона библиотекой `xhtml2pdf`.

Целевая операционная система — Windows 10 и Windows 11.

Не использовать:

- Chromium и другие браузеры;
- Playwright, Selenium и WebDriver;
- Microsoft Word, `WINWORD.EXE`, COM и `pywin32`;
- LibreOffice, `wkhtmltopdf` и внешние PDF-конвертеры;
- поставляемые вместе с проектом файлы шрифтов.

Для русского текста использовать системный шрифт Arial, входящий в состав Windows.

Актуальный исходный шаблон:

```text
templates/refund_application_2_xhtml2pdf.html
```

Программа должна использовать этот HTML-файл как единственный шаблон заявления. Конвертация шаблона из DOC, DOCX или Markdown не требуется.

## 2. Общий сценарий

1. Пользователь запускает скрипт, передавая путь к JSON-файлу возврата.
2. Скрипт читает `config.ini` из каталога, в котором расположен сам скрипт.
3. Скрипт проверяет конфигурацию и создаёт отсутствующие каталоги логов и результатов.
4. Скрипт читает и проверяет JSON.
5. Скрипт загружает актуальный HTML-шаблон.
6. Скрипт рассчитывает общую сумму возврата.
7. Значения безопасно подставляются в метки шаблона.
8. Заполненный HTML преобразуется в PDF библиотекой `xhtml2pdf`.
9. PDF сохраняется в каталог из `config.ini`.
10. В консоль выводится полный путь к созданному PDF.
11. PDF открывается назначенной в Windows программой просмотра.

Дополнительная форма ввода перед созданием заявления не требуется.

## 4. Входные данные

Скрипт должен принимать путь к одному JSON-файлу.

Пример структуры:

```json
{
  "date": "20260720",
  "org": "ООО \"Клевер Урал\"",
  "recipient": "Генеральному директору ООО \"Клевер Урал\" Валейня И. Л.",
  "address": "Тольятти, Автозаводское шоссе, 6",
  "items": [
    {
      "name": "Наименование товара 1",
      "price": 2932
    },
    {
      "name": "Наименование товара 2",
      "price": 1732
    }
  ]
}
```

Поддерживаемые кодировки JSON:

- UTF-8;
- UTF-8 с BOM;
- Windows-1251.

## 5. Метки актуального шаблона

Шаблон содержит только следующие метки:

| Метка | Источник | Правило |
| --- | --- | --- |
| `{recipient}` | `recipient` | Подставить получателя заявления |
| `{org}` | `org` | Подставить организацию во все места, где встречается метка |
| `{address}` | `address` | Подставить адрес магазина |
| `{date}` | `date` | Преобразовать дату покупки из `YYYYMMDD` в `DD.MM.YYYY` |
| `{items}` | `items[].name` | Сформировать безопасный HTML-фрагмент со строками товаров |
| `{total_amount}` | `items[].price` | Подставить сумму всех цен в целых рублях |
| `{application_date}` | системная дата | Дата создания заявления в формате `DD.MM.YYYY` |

Все метки должны иметь формат `{field_name}` с одной парой фигурных скобок. Метки с двойными фигурными скобками не использовать.

Поле `date` — дата покупки. Оно не должно использоваться как дата формирования заявления.

Перед созданием PDF проверить, что:

- в шаблоне присутствуют все семь обязательных меток;
- неизвестных меток нет;
- после подстановки не осталось незаполненных служебных меток.

Одна и та же разрешённая метка может встречаться в шаблоне несколько раз. В частности, все вхождения `{org}` должны быть заменены.

## 6. Вывод товаров и безопасная подстановка

Все значения, полученные из JSON, необходимо экранировать через `html.escape(value, quote=True)` до включения в HTML.

Обычные метки (`recipient`, `org`, `address`, `date`, `total_amount`, `application_date`) подставляются как экранированный текст.

Для `{items}` применяется отдельное правило:

1. Каждое `items[].name` экранировать отдельно.
2. Если товар один, вывести его без номера.
3. Если товаров несколько, сохранить исходный порядок и добавить нумерацию, начиная с 1.
4. Каждую строку поместить в созданный программой элемент `<div class="item">...</div>`.
5. Не экранировать повторно сами элементы `<div>`, сформированные программой.
6. HTML из названия товара не должен интерпретироваться как разметка.

Пример для двух товаров:

```html
<div class="item">1. Первый товар</div>
<div class="item">2. Второй товар</div>
```

Обычную строку с товарами, разделёнными символами `\n`, не использовать: актуальный шаблон не зависит от браузерного свойства `white-space: pre-line`.

Из каждой позиции брать только `name`. Артикул, код, количество и цену отдельной позиции в заявление не выводить.

## 7. Расчёт суммы

Общую сумму вычислять только по значениям `items[].price`:

```text
total_amount = сумма всех items[i].price
```

Требования:

- цена должна быть неотрицательным целым числом;
- логическое значение `true`/`false` не считать числом;
- дробные значения не принимать;
- итоговая сумма должна быть больше нуля;
- разряды суммы разделять пробелом: `7 596`;
- в `{total_amount}` передавать только число без слова «рублей».

Слова «рублей» и «копеек» уже находятся в HTML-шаблоне. Поле копеек остаётся пустой линией для ручного заполнения.

## 8. Требования к HTML-шаблону

Использовать актуальный файл `refund_application_2_xhtml2pdf.html` без изменения во время работы программы.

Шаблон уже адаптирован под `xhtml2pdf` и содержит:

- формат страницы A4 и поля через `@page`;
- табличную разметку правой шапки, растягиваемых линий и подписей;
- линии, доходящие до правого поля в блоках после «от» и после «Я,»;
- простой CSS без Grid, Flexbox и JavaScript;
- метки, перечисленные в разделе 5;
- ссылки на системные Arial Regular и Arial Bold.

Не заменять растягиваемые линии фиксированным количеством символов `_`. Они должны оставаться нижними границами ячеек таблиц, чтобы доходить до правого края доступной области.

Запрещено без отдельного согласования:

- возвращать CSS Grid или Flexbox;
- добавлять JavaScript;
- добавлять сетевые изображения, стили или шрифты;
- менять текст заявления и юридические формулировки;
- менять поля страницы, размеры шрифта и табличную геометрию;
- возвращать зависимость от Chromium.

Заполненный HTML формировать в памяти. Исходный шаблон не перезаписывать и не создавать постоянную заполненную HTML-копию.

## 9. Системный шрифт Arial

Шаблон использует стандартные файлы Windows:

```text
C:\Windows\Fonts\arial.ttf
C:\Windows\Fonts\arialbd.ttf
```

В HTML они подключены как URI:

```text
file:///C:/Windows/Fonts/arial.ttf
file:///C:/Windows/Fonts/arialbd.ttf
```

До формирования PDF программа обязана проверить наличие обоих файлов.

Если обычный или полужирный Arial отсутствует либо недоступен для чтения:

- PDF не создавать;
- вывести понятную ошибку на русском языке с полным путём к отсутствующему файлу;
- записать техническую ошибку в лог.

Молча заменять Arial на Helvetica или другой встроенный шрифт запрещено, поскольку это может повредить кириллицу и изменить вёрстку.

2. Пути к шаблону, логам и результатам не хранить в Python-коде.
3. Пути к Arial отдельно в `config.ini` не добавлять: использовать стандартные пути Windows, заданные актуальным шаблоном и проверяемые программой.
4. Отсутствующие каталоги логов и PDF создавать автоматически.
5. При отсутствии `config.ini`, секции `[paths]` или обязательного параметра завершать работу с понятной ошибкой.
6. При отсутствии HTML-шаблона PDF не создавать.

## 12. Формирование PDF

Рекомендуемый интерфейс:

```python
def create_pdf(document_html: str, pdf_path: Path, template_path: Path) -> None:
    ...
```

Функция должна:

1. Создать родительский каталог результата, если он отсутствует.
2. Открыть PDF в бинарном режиме `wb`.
3. Вызвать `xhtml2pdf.pisa.CreatePDF()` с UTF-8 HTML.
4. Передать базовый путь исходного HTML-шаблона.
5. Передать функцию разрешения локальных ресурсов `link_callback`.
6. Проверить `status.err`.
7. Проверить существование и ненулевой размер файла.
8. Проверить сигнатуру `%PDF-`.
9. При любой ошибке удалить частично созданный PDF.
10. Преобразовать техническую ошибку в понятный пользователю `ApplicationError`.

Пример вызова:

```python
status = pisa.CreatePDF(
    src=document_html,
    dest=pdf_file,
    encoding="utf-8",
    path=template_path.resolve().as_uri(),
    link_callback=link_callback,
)
```

Конкретный вариант параметра `path` должен соответствовать закреплённой версии `xhtml2pdf` и быть покрыт интеграционным тестом.

### Правила `link_callback`

Разрешить только:

- локальный исходный HTML-шаблон и ресурсы внутри каталога проекта, если они появятся по согласованному ТЗ;
- два точных системных URI Arial:
  - `file:///C:/Windows/Fonts/arial.ttf`;
  - `file:///C:/Windows/Fonts/arialbd.ttf`.

Для Arial возвращать соответствующие абсолютные Windows-пути.

Запретить:

- `http://`, `https://` и любые сетевые URI;
- UNC-пути;
- произвольные `file://` URI;
- выход из каталога проекта через `..`;
- любые файлы из `C:\Windows\Fonts`, кроме `arial.ttf` и `arialbd.ttf`.

Сравнение разрешённых путей на Windows выполнять без учёта регистра и после нормализации пути.

## 13. Проверка входных данных

Перед генерацией проверить:

- JSON-файл существует и читается;
- содержимое является корректным JSON-объектом;
- поля `recipient`, `org`, `address`, `date` существуют и являются непустыми строками;
- `date` является реальной датой формата `YYYYMMDD`;
- `items` является непустым массивом;
- каждый элемент `items` является объектом;
- `items[].name` является непустой строкой;
- `items[].price` является неотрицательным целым числом;
- итоговая сумма больше нуля;
- конфигурация и шаблон доступны;
- системные `arial.ttf` и `arialbd.ttf` существуют и читаются;
- заполненный шаблон не содержит незаменённых меток;
- преобразование завершилось без ошибок;
- результат является непустым PDF.

Повреждённый или неполный JSON не должен приводить к созданию частично заполненного заявления.

## 14. Имя и открытие результата

Имя PDF:

```text
Заявление_на_возврат_<имя_JSON>_<YYYY-MM-DD_HH-MM-SS>.pdf
```

Если файл с таким именем уже существует, добавить числовой суффикс. Ранее созданные файлы не перезаписывать.

После успешного создания открыть файл через `os.startfile()`.

Если PDF создан, но автоматическое открытие не удалось:

- не удалять PDF;
- вывести полный путь к готовому файлу;
- отдельно сообщить об ошибке открытия;
- вернуть ненулевой код завершения либо иной явно задокументированный код частичного успеха.
